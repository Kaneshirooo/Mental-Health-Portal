Part 2: Database Reconciliation Log

Earlier, we performed a Normalization process to align the legacy mental_health_db with modern Laravel standards.

  • Consistent Piling: Converted camelCase legacy columns to snake_case.
  • UUID vs BigInt: Standardized on bigint for primary keys in transactional tables for performance.
  • Soft Deletes: Added support for deleted timestamps to allow data recovery.
  • Role Management: Migrated from loose strings to Enums for user_type.

Part 3: Determining Relationships

To identify how any two tables in this project are linked, follow this hierarchy:

1. The Migration File (Physical)

Foreign keys are established here. If a table has a foreignId, it physically depends on the parent table.

$table->foreignId('student_id')->constrained('users', 'user_id');

2. The Eloquent Model (Logical)

Laravel models use these methods to fetch related data automatically:

MethodUsageProject Example
belongsTo()N:1 RelationshipAppointment → User

Part 4: Roles & Permissions (Spatie Logic)

You asked about model_has_roles and model_has_permissions. These implement Advanced Access Control.

  • Flexible Membership: One user can hold multiple roles simultaneously.
  • Polymorphic Linking: The model_type column allows roles to be assigned to Users, Admins, or even Teams.
  • Granular Override: Permissions can be granted to an individual user even if their role doesn't normally have it.
Implementation Note
While your project currently uses a user_type column for simplicity, the Spatie tables provide the infrastructure for a future-proof permission system as the portal grows.
